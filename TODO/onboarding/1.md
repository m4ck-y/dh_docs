## 🧩 Arquitectura de Waitlist, Applicant y User

## 🧠 Modelo general

El sistema se divide en 3 entidades principales:

1. **Waitlist (opcional)** → Leads o interesados
2. **Applicant (obligatorio)** → Usuario en proceso de onboarding
3. **User (final)** → Usuario aprobado y activo

<aside>
⚠️

**Regla clave:** todos los usuarios **deben** pasar por **Applicant** antes de convertirse en **User**.

</aside>

---

## 🔄 Flujo general

```
(waitlist opcional) → invite → applicant (onboarding) → review → user
```

---

## 📥 1. Waitlist (opcional)

Se usa solo si quieres controlar acceso anticipado.

### Estructura

```json
{
  "email": "string",
  "status": "pending | invited | converted",
  "created_at": "datetime"
}
```

---

## ✉️ 2. Invitación

Se genera un token para permitir acceso al onboarding.

```json
{
  "email": "string",
  "token": "string",
  "expires_at": "datetime",
  "used": false
}
```

---

## 👤 3. Applicant (onboarding obligatorio)

Aquí vive todo el proceso de registro en **4 pasos**.

<aside>
❗

Importante: aquí aún **no** existe un usuario del sistema.

</aside>

### 📊 Estructura del Applicant

```json
{
  "id": "uuid",
  "email": "string",
  "status": "in_progress | submitted | under_review | approved | rejected",
  "onboarding_step": 1,
  "personal_info": {},
  "contact_verified": false,
  "password_hash": null,
  "documents": [],
  "created_at": "datetime",
  "submitted_at": "datetime"
}
```

### 🪜 Pasos del onboarding

#### 1) Datos personales

- Se guardan datos básicos
- Avanza a step 2

#### 2) Verificación de contacto (OTP)

- Se envía código OTP
- Se valida

Actualización esperada:

```json
{ "contact_verified": true }
```

#### 3) Creación de contraseña

- Se guarda `password_hash`
- Aún **no** es usuario

#### 4) Carga de documentación

- Se suben archivos requeridos
- Se cambia el estado a:

```json
{ "status": "submitted" }
```

---

## 🔍 4. Evaluación

Puede ser manual o automática.

Lógica sugerida:

```
if cumple_requisitos:
  status = "approved"
else:
  status = "rejected"
```

Estados recomendados:

```
in_progress → submitted → under_review → approved / rejected
```

---

## 👤 5. Creación del usuario

✅ **Solo ocurre si el applicant es aprobado**.

```json
{
  "email": "string",
  "password_hash": "string",
  "created_at": "datetime",
  "verified": true
}
```

Relación:

```
applicant → user_id
```

---

## ⚠️ Reglas críticas

### ❌ No hacer

- Crear usuarios antes de aprobación
- Saltarse el Applicant
- No validar OTP
- No expirar tokens

### ✅ Buenas prácticas

#### 🔐 Seguridad

- OTP con expiración
- Tokens de invitación únicos
- Hash de contraseña (bcrypt/argon2)

#### ⏱️ Expiraciones

```json
{
  "otp_expires_at": "datetime",
  "invite_expires_at": "datetime",
  "onboarding_expires_at": "datetime"
}
```

#### 🔄 Reintentos

- Permitir subir documentos otra vez si fue rechazado

#### 💾 Persistencia de progreso

- Guardar `onboarding_step`
- Permitir retomar proceso

---

## 🧾 Resumen

- Waitlist es opcional
- Applicant es obligatorio
- User solo existe tras aprobación

```
Applicant ≠ User
```

El Applicant es un estado previo, no un usuario incompleto.

---

## 🚀 Flujo final simplificado

1. (Opcional) Waitlist
2. Invitación con token
3. Applicant inicia onboarding
4. Completa 4 pasos
5. Evaluación
6. Aprobado → se crea User

```jsx
IF product_state = building:
    user → waitlist

IF product_state = live:
    user → applicant

IF user exists in waitlist AND product becomes live:
    send invite → applicant

invite = optional access gate
```
