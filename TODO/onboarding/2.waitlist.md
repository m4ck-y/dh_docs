# 📘 WAITLIST API DESIGN (v1)

## 📌 Base Path

```
/api/v1/waitlist
```

---

# 🧠 1. Concepto

La waitlist representa un:

> 🔹 Lead interesado en el producto antes del onboarding
> 

No es usuario, no es applicant.

---

# 👤 2. WAITLIST ENTITY (modelo)

## 📦 Schema principal

```
{
  "id":"uuid",
  "client_name":"string",
  "email":"string",
  "status":"string",
  "source":"string",
  "created_at":"datetime",
  "updated_at":"datetime",
  "invited_at":"datetime | null",
  "converted_at":"datetime | null"
}
```

---

## 🧠 Campos explicados

### 👤 client_name

```
Full name del lead (first_name + last_name)
```

Ejemplo:

```
"client_name":"John Doe"
```

---

### 📧 email

```
Identificador único del lead
```

👉 Debe ser UNIQUE

---

### 📊 status

Estado del lead en el sistema

---

### 📡 source

Opcional pero MUY recomendado:

```
waitlist_form
landing_page
referral
ads
partner
```

---

# 📊 3. WAITLIST STATUSES

## 🔥 Estados recomendados (completos y reales)

```
pending
active
invited
converted
expired
blocked
```

---

## 🧠 Significado

### 🟡 pending

```
Lead recién creado
```

---

### 🟢 active

```
Está en waitlist esperando acceso
```

---

### ✉️ invited

```
Se le envió invitación a onboarding
```

---

### 🔁 converted

```
Ya pasó a applicant
```

---

### ⛔ expired

```
Lead ya no es válido (muy viejo o abandonado)
```

---

### 🚫 blocked

```
Spam / abuso / inválido
```

---

# 📋 4. API ENDPOINTS

---

# ➕ 1. Crear waitlist entry

```
POST /api/v1/waitlist
```

---

## Request

```
{
  "client_name":"John Doe",
  "email":"john@email.com",
  "source":"landing_page"
}
```

---

## Response

```
{
  "id":"uuid",
  "client_name":"John Doe",
  "email":"john@email.com",
  "status":"pending",
  "created_at":"datetime"
}
```

---

# 📄 2. Obtener lead

```
GET /api/v1/waitlist/{email}
```

---

# 📋 3. Listar leads

```
GET /api/v1/waitlist?status=active
```

---

# ✉️ 4. Invitar lead (trigger onboarding)

```
POST /api/v1/waitlist/{email}/invite
```

---

## Behavior

- status → `invited`
- genera invite token
- envía email

---

# 🔁 5. Convertir a applicant

```
POST /api/v1/waitlist/{email}/convert
```

---

## Behavior

```
waitlist → converted
create applicant
link applicant_id
```

---

# ❌ 6. Bloquear lead

```
POST /api/v1/waitlist/{email}/block
```

---

# ⏳ 7. Expirar leads (batch job)

```
POST /api/v1/waitlist/expire
```

---

# 🧠 5. REGLAS DE NEGOCIO

---

## 🔐 Email único

```
email must be unique
```

---

## 🔁 No duplicación

Si existe:

```
status != blocked
```

👉 no crear nuevo registro

---

## ✉️ Invite logic

```
only active → can be invited
```

---

## 🔄 Convert logic

```
only invited OR active → can convert
```

---

# 🧩 6. RELACIÓN CON APPLICANT

```
waitlist.email ↔ applicant.email
waitlist.id → applicant.waitlist_id
```

---

## Flujo

```
WAITLIST (lead)
   ↓ invite
APPLICANT (onboarding)
   ↓ submit
REVIEW
   ↓ approve
USER
```
