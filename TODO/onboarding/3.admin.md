# 📘 ADMIN API DESIGN (v1)

## 📌 Base Path

```
/api/v1/admin
```

---

# 🧠 1. Concepto

El Admin API es responsable de:

> 🔥 controlar acceso, evaluar applicants y convertir leads (waitlist → onboarding)
> 

---

# 🧩 2. DOMINIOS DEL ADMIN

```
1. Waitlist management
2. Applicant review
3. Decision engine (approve / reject)
4. Invitation system
5. Document review
6. Analytics (optional)
```

---

# 👥 3. WAITLIST (ADMIN CONTROL)

---

## 📋 List waitlist leads

```
GET /api/v1/admin/waitlist
```

---

## 🔍 Get lead detail

```
GET /api/v1/admin/waitlist/{email}
```

---

## ✉️ Invite waitlist user (clave de tu sistema)

```
POST /api/v1/admin/waitlist/{email}/invite
```

---

## 🧠 Behavior

```
1. check if lead exists
2. generate invite token
3. mark status = invited
4. send email
5. allow onboarding start
```

---

## 🔁 Bulk invite (growth feature)

```
POST /api/v1/admin/waitlist/invite-bulk
```

```
{
  "emails": ["a@mail.com","b@mail.com"]
}
```

---

## 🚫 Block lead

```
POST /api/v1/admin/waitlist/{email}/block
```

---

# 👤 4. APPLICANT REVIEW (CORE SYSTEM)

---

## 📋 List applicants

```
GET /api/v1/admin/applicants?status=under_review
```

---

## 🔍 Get applicant detail

```
GET /api/v1/admin/applicants/{id}
```

---

## 🧠 Approve applicant (CRÍTICO)

```
POST /api/v1/admin/applicants/{id}/approve
```

---

### Behavior

```
1. validate applicant = submitted / under_review
2. set status = approved
3. create user
4. link user_id ↔ applicant_id
5. send email "you're approved"
```

---

## ❌ Reject applicant

```
POST /api/v1/admin/applicants/{id}/reject
```

```
{
  "reason":"invalid_documents | failed_verification | risk_detected",
  "notes":"optional admin comment"
}
```

---

### Behavior

```
1. set status = rejected
2. store reason
3. notify user
```

---

## 🔁 Request more info

```
POST /api/v1/admin/applicants/{id}/request-info
```

---

### Behavior

```
status → in_progress
current_step → documents or personal_info
```

---

# ✉️ 6. INVITE → APPLICANT FLOW (CLAVE DEL SISTEMA)

Este es tu caso más importante.

---

## 🧠 Invite waitlist user

```
POST /api/v1/admin/waitlist/{email}/invite
```

---

### Flujo completo

```
WAITLIST USER
   ↓ (admin invite)
INVITE GENERATED
   ↓
USER CAN START ONBOARDING
   ↓
APPLICANT CREATED
```

---

## 🧾 Invite payload (interno)

```
{
  "email":"user@email.com",
  "token":"secure_random_token",
  "expires_at":"datetime"
}
```

---

## 🔐 Validate invite

```
POST /api/v1/invites/validate
```

# 🧠 8. RULES DEL ADMIN SYSTEM

---

## ❌ No permitir

- aprobar sin submitted
- crear user manualmente sin applicant
- saltarse documents si son obligatorios
- invitar emails bloqueados

---

## ✅ Permitir

- invite desde waitlist
- bulk invite
- request-info (reabrir applicant)
- document-level rejection
- manual override (solo admin root)

---
