# 📘 APPLICANT API DESIGN (v1)

## 📌 Base Path

```
/api/v1/onboarding
```

👉 (Recomendado: “onboarding” en vez de “applicant” porque es más product/business oriented)

---

# 🧠 1. Concepto

El Applicant es:

> 🔥 Usuario en proceso de validación + onboarding + evaluación
> 

No es usuario final.

---

# 👤 2. APPLICANT ENTITY (schema completo)

## 📦 Schema principal

```
{
  "id":"uuid",
  "email":"string",

  "status":"string",
  "current_step":"string",

  "personal_info": {
    "first_name":"string",
    "last_name":"string",
    "country":"string"
  },

  "contact_verification": {
    "email_verified":"boolean",
    "phone_verified":"boolean",
    "otp_attempts":"number",
    "otp_expires_at":"datetime"
  },

  "password_hash":"string | null",

  "documents": [
    {
      "id":"uuid",
      "type":"string",
      "url":"string",
      "status":"string",
      "rejection_reason":"string | null"
    }
  ],

  "evaluation": {
    "score":"number | null",
    "review_status":"string",
    "reviewed_by":"string | null"
  },

  "waitlist_id":"uuid | null",

  "created_at":"datetime",
  "updated_at":"datetime",
  "submitted_at":"datetime | null",
  "approved_at":"datetime | null",
  "rejected_at":"datetime | null"
}
```

# 📊 3. APPLICANT STATUSES

## 🔥 Estados principales

```
in_progress
submitted
under_review
approved
rejected
blocked
expired
abandoned
```

---

## 🧠 Significado

### 🟡 in_progress

```
Usuario aún completando onboarding
```

---

### 🟢 submitted

```
Terminó onboarding y envió a revisión
```

---

### 🔍 under_review

```
Está siendo evaluado (manual o automático)
```

---

### ✅ approved

```
Pasó evaluación → se crea User
```

---

### ❌ rejected

```
No pasó evaluación
```

---

### ⛔ blocked

```
Fraude / abuso / inválido
```

---

### ⏳ expired

```
No completó onboarding en tiempo
```

---

### 💨 abandoned

```
Abandonó el flujo sin terminar pasos
```

# 📋 5. API ENDPOINTS (CORE)

---

# 🚀 1. Crear applicant (start onboarding)

```
POST /api/v1/onboarding/start
```

---

## Body

```
{
  "email":"user@email.com",
  "waitlist_id":"uuid | null",
  "invite_token":"string | null"
}
```

---

## Response

```
{
  "applicant_id":"uuid",
  "status":"in_progress",
  "current_step":"personal_info"
}
```

---

# 👤 2. Get applicant state

```
GET /api/v1/onboarding/{applicant_id}
```

---

# 🧠 3. PERSONAL INFO STEP

```
POST /api/v1/onboarding/{id}/personal-info
```

---

## Body

```
{
  "first_name":"John",
  "last_name":"Doe",
  "country":"MX"
}
```

---

# 📧 4. OTP STEP

```
POST /api/v1/onboarding/{id}/otp/send
POST /api/v1/onboarding/{id}/otp/verify
```

---

## Verify body

```
{
  "code":"123456"
}
```

---

# 🔐 5. PASSWORD STEP

```
POST /api/v1/onboarding/{id}/password
```

---

## Body

```
{
  "password":"plain-text"
}
```

---

# 📄 6. DOCUMENTS STEP

```
POST /api/v1/onboarding/{id}/documents
```

---

## Multipart upload:

- file
- type: id | proof_of_address | selfie

---

# 📤 7. SUBMIT FOR EVALUATION

```
POST /api/v1/onboarding/{id}/submit
```

---

## Behavior

```
status → submitted
current_step → null
trigger evaluation pipeline
```

---

# 🔍 8. EVALUATION STATUS

```
GET /api/v1/onboarding/{id}/status
```

---

## Response

```
{
  "status":"under_review",
  "score":87,
  "current_step":null
}
```

---

# 🧪 9. RE-SUBMIT (si rejected)

```
POST /api/v1/onboarding/{id}/resubmit
```

---

# 🧠 6. EVALUATION ENGINE (concepto)

```
rules engine:
- documents valid?
- OTP verified?
- risk score
- duplicate detection
```

# 🔴 7. ADMIN API (APPLICANT REVIEW)

---

## List applicants

```
GET /api/v1/admin/applicants?status=under_review
```

---

## Get applicant

```
GET /api/v1/admin/applicants/{id}
```

---

## Approve

```
POST /api/v1/admin/applicants/{id}/approve
```

---

## Reject

```
POST /api/v1/admin/applicants/{id}/reject
```

```
{
  "reason":"invalid_documents"
}
```

---

## Request changes

```
POST /api/v1/admin/applicants/{id}/request-info
```

---

## Document review

```
POST /api/v1/admin/applicants/{id}/documents/{doc_id}/approve
POST /api/v1/admin/applicants/{id}/documents/{doc_id}/reject
```

# 🔐 9. REGLAS CRÍTICAS

## ❌ No hacer

- crear user antes de approve
- saltar steps
- modificar status desde frontend
- mezclar waitlist logic aquí

---

## ✅ Sí hacer

- state machine controlado por backend
- steps explícitos
- evaluation separada
- idempotent endpoints
