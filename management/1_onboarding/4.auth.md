# 4. Auth

Capa de autenticación e identidad. Servicio responsable: `dh_auth` (ver TASK-004).

## Separación de conceptos

> **Google Login ≠ MFA.**
> Google OAuth/OpenID Connect es un **proveedor de identidad externo** — va en `dh_auth`.
> MFA (TOTP, WebAuthn, SMS/email como segundo factor) es un mecanismo de verificación adicional — va en `dh_mfa` (futuro, TASK-006).

| Servicio | Qué maneja |
|---|---|
| `dh_auth` | Login tradicional, Google OAuth, emisión de JWT, refresh, logout, sesiones |
| `dh_mfa` *(futuro)* | TOTP (Google Authenticator), WebAuthn, biometría, políticas de segundo factor |

### Estrategia de Sesión (Silent Refresh)
El sistema implementa **Silent Refresh at Gateway Level** (ver [ADR 019](../../decisions/019-silent-refresh-gateway-level.md)):
- **Transporte**: Cookies **HttpOnly**, `Secure` y `SameSite=Lax`.
- **Transparencia**: El `api_middleware` renueva el `access_token` automáticamente usando el `refresh_token` sin intervención del frontend.
- **Seguridad**: Los JWT se firman con RSA/ECDSA; la clave pública está disponible para validación stateless.

## 4.1 Login tradicional

```
POST /v1/auth/login
Servicio: dh_auth
```

**Body:**
```json
{
  "username": "juan.perez",
  "password": "MySecurePass123!"
}
```

**Respuesta exitosa:**
```json
{
  "message": "Login successful",
  "user": {
    "email": "juan.perez"
  }
}
```
*Nota: Los tokens JWT se emiten vía cookies **HttpOnly** (`access_token`, `refresh_token`).*

**Si MFA está habilitado (segundo factor configurado en dh_mfa):**
```json
{
  "data": {
    "mfa_required": true,
    "challenge_id": "uuid"
  }
}
```

---

## 4.2 Login con Google (OAuth / OpenID Connect)

```
GET /v1/auth/google
Servicio: dh_auth
```

Redirige al flujo OAuth de Google. Al completar, Google regresa al callback:

```
GET /v1/auth/google/callback?code=...
Servicio: dh_auth
```

`dh_auth` intercambia el código por el token de Google, obtiene el perfil del usuario y emite el JWT propio del sistema.

---

## 4.3 Refresh de token

```
POST /v1/auth/refresh
Servicio: dh_auth
Body: { "refresh_token": "eyJ..." }
```

---

## 4.4 Logout

```
POST /v1/auth/logout
Servicio: dh_auth
Headers: Authorization: Bearer {access_token}
```

Revoca el refresh token activo.

## 4.6 Claves públicas (verificación de JWT)

```
GET /v1/auth/.well-known/jwks.json
Servicio: dh_auth
```

Otros microservicios usan este endpoint para obtener la clave pública y verificar tokens sin llamar a `dh_auth` en cada request.

---

## 4.7 Recuperación de contraseña

```
POST /v1/auth/forgot-password
Body: { "email": "juan@example.com" }
```

PulseCore envía el código OTP vía email/SMS (propósito `RESET_PASSWORD`).

```
POST /v1/auth/reset-password
Body: { "email": "juan@example.com", "code": "B2C3D4", "new_password": "NuevoPass456!" }
```

## Códigos de error

| HTTP | Código interno | Significado |
|---|---|---|
| 401 | 10 | Sesión no iniciada |
| 401 | 11 | Token expirado |
| 401 | 12 | Token inválido |
| 401 | 13 | Usuario bloqueado |
| 401 | 14 | Refresh token inválido o revocado |
| 403 | 20 | Sin permisos para este recurso |
| 400 | 30 | Email inválido |
| 400 | 31 | Contraseña débil |
| 400 | 32 | OTP incorrecto |
